Angular Wrapper

Standalone Angular integration for the shared primitive layer.

The Angular wrapper keeps product teams inside a familiar Angular template model while the shared Web Component core continues to own behavior, accessibility, and public runtime semantics.

UI kit reference

Keep Angular ergonomics while sharing one logic-only component core.

@fuggetlenfe/angular-wrapper exports standalone Angular-ready directives that map to the shared Stencil primitives. The wrapper does not decide the active brand, theme, or surface styling. Those remain in the consumer app shell and in the external styling packages.

Getting started

Angular consumers import the standalone wrapper exports into their component and pull the shared token contract plus the selected brand stylesheets into the application styles.

pnpm add @fuggetlenfe/angular-wrapper @fuggetlenfe/tokens @fuggetlenfe/brand-styles
import { Component } from '@angular/core'; import { FfButton } from '@fuggetlenfe/angular-wrapper'; @Component({ selector: 'app-root', imports: [FfButton], templateUrl: './app.html' }) export class AppComponent {}
@import '@fuggetlenfe/tokens/contract.css'; @import '@fuggetlenfe/brand-styles/brand-1-light.css'; @import '@fuggetlenfe/brand-styles/brand-1-dark.css';

Architecture boundaries

The Angular wrapper exists so Angular teams can consume the shared primitive layer through standalone imports and template bindings. It should not grow into a second design system or a place for application-specific decisions.

What belongs here

Standalone Angular exports generated from the shared Stencil core.
Property and event bridge behavior for Angular templates.
Typed integration surface for Angular product teams.

What does not belong here

Theme activation or brand ownership.
Business workflows or page-level layout rules.
Any hardcoded styling that should live in the brand library.

Component reference

The Angular wrapper preserves the same public primitive API that exists in the Stencil layer. The PoC is intentionally limited to the button because that is the primitive currently represented in Figma and covered by the sync contract.

Primitive Standalone import

FfButton

Shared action primitive for shell actions, content actions, and theme demonstrations. The Angular wrapper exposes it in a template-friendly form without injecting design logic.

<main data-brand="brand-1" [attr.data-theme]="activeTheme()"> <ff-button type="submit">Save changes</ff-button> </main>
PropTypeDescription
labelstringFallback label when no projected content is provided.
disabledbooleanDisables pointer and keyboard interaction.
fullWidthbooleanExpands the host without coupling layout to a specific brand.
type'button' | 'submit' | 'reset'Mirrors the native button type attribute.

Theming and brand activation

Angular applications activate styling by importing the token contract and the official brand pack, then setting the visual context on the outer shell with data-brand and data-theme.

<main class="consumer-shell" data-brand="brand-1" [attr.data-theme]="activeTheme()"> <ff-button>Launch selected brand</ff-button> </main>

Supported override paths

Application shell data-brand and data-theme context.
Documented CSS custom properties from the shared token contract.
Host classes passed from the Angular template.

Accessibility expectations

Accessibility behavior belongs to the shared primitive layer, but the Angular wrapper must preserve that contract through template bindings and event handling.

AreaCurrent expectation
Keyboard supportButtons keep native button semantics. Dropdown supports Arrow keys, Enter, Escape, and Tab.
ARIA modelDropdown exposes a listbox pattern with proper selected and expanded state through the primitive.
Theme qualityOfficial brand packs are expected to meet WCAG AA contrast targets before release.

Release and maintenance model

The wrapper is intended to ship as part of a centrally versioned platform. Product teams should consume it as a package. Shadcn-style copy ownership may exist only as an escape hatch for exceptional product requirements, not as the default path.

Recommended enterprise policy

Default: consume the official wrapper package and official brand styles.
Exception: copy ownership only with explicit ownership and upgrade responsibility.
Never fork the official corporate brand pack as the default delivery model.
